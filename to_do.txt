## Task 1 — Fix Identical File Detection (Gap Issue)

Files to read first: engine/api.py, engine/evidence/evidence.py

The problem:
When two submissions are completely identical, the engine does not flag the entire
file. It flags scattered regions with gaps in between. For example lines 1-30 are
flagged, then lines 31-50 are not, then 51-80 are flagged again — even though the
gap lines are also identical. This gives professors a false impression that only
parts of the submission match.

Why this happens:
Winnowing deliberately subsamples fingerprints — it selects roughly 1 in every W
fingerprints from the token stream. This means not every line of code produces a
fingerprint that gets compared. For low similarity files this is fine. For identical
or near-identical files, this subsampling leaves gaps in the matched regions. These
gaps appear as unhighlighted lines in the full code view even though the actual code
is copied.

There are two complementary fixes needed:

Fix 1 — merge_gap:
The engine has a parameter called merge_gap that controls how far apart two matched
regions can be (in lines) before they are treated as separate blocks rather than
merged into one. It is currently hardcoded at 3 lines everywhere. For high similarity
files this is too small — the Winnowing gaps are larger than 3 lines. The fix is to
make merge_gap scale with the similarity score. The higher the score, the larger
merge_gap should be, so nearby matched regions are merged across the Winnowing gaps.
This must only affect how evidence blocks are merged together — it must not change
the scoring logic, the fingerprinting, or any other part of the pipeline.

Fix 2 — full file highlight for near-identical submissions:
Even with a larger merge_gap, some gaps may persist for 100% identical files.
At very high similarity scores (95% and above), the correct interpretation is that
the entire file is a match. The engine should detect this case and mark the entire
file as highlighted rather than only the fingerprint-matched regions. There is already
a function in evidence.py called _compute_line_highlights() that returns meaningful
line numbers from a block of source code, excluding comment-only lines and structural
brace lines. This same function should be used to compute highlights for the full file
in this case. The result should be stored in line_highlights_a and line_highlights_b
and a flag called identicalSubmissions should be added to the result to let the
frontend know to treat this as a full-file match.

What must not change:
- The scoring logic (weighted_score) must not be touched
- The fingerprinting (kgrams/winnow) must not be touched
- The tokenizer must not be touched
- The comment stripping and canonicalization must not be touched
- The evidence block structure (file names, line ranges, code snippets, match strength)
  must remain the same — only line_highlights_a/b at the top level are overwritten
  for the identical case

## Task 2 — Compare-All Round Robin, Scores Only

Files to read first: engine/api.py, routes_engine.py

The current state:
The engine has a function called batch_analyze() that compares all submissions
against each other (all pairs). Currently it runs the full pipeline for every pair —
this includes building evidence blocks, running obfuscation detection, and filtering
out pairs below a similarity threshold. Only pairs above the threshold are returned.
The backend route /compare-all calls this function and stores results to the database.

The problem:
This design is slow and incomplete. Running full evidence building for every pair is
expensive and wasteful — the professor has not asked for detailed evidence yet, they
just want to see a ranked list of all pairs. The threshold filter makes it worse by
silently discarding pairs the professor might want to see. A pair at 35% similarity
might still be suspicious and the professor should be able to see it.

What needs to change:
batch_analyze() should become a lightweight scoring-only pass. Its only job is to
compute a similarity score for every possible pair of submissions and return them all
ranked from highest to lowest. It should not build evidence blocks. It should not run
obfuscation detection. It should not filter anything out. Every pair gets a score and
every pair is returned.

The detailed work — evidence blocks, line highlights, obfuscation flags — happens
later when the professor selects a specific pair from the list. That is handled by
the existing compare() function which already does the full pipeline correctly.

There is also a group feature needed. Submissions can optionally belong to a group.
If two submissions are in the same group they should not be compared against each
other. For example a professor might upload reference files tagged as one group and
student submissions as another. The engine should skip any pair where both submissions
carry the same group tag. This is the same pattern as the existing skip_same_student
logic already in batch_analyze() — follow that same approach.

The backend route /compare-all also needs to be updated to match the new output
shape from batch_analyze(). Since batch no longer builds evidence, the route should
not try to store evidence rows to the database. It should only store a score row per
pair. The return to the frontend should be the full ranked list of all pairs with
their scores — no threshold filtering, no evidence.

What must not change:
- compare() must not be touched for this task
- The scoring logic must not change — only the surrounding pipeline is being stripped
- The database schema does not change — only what gets written to it changes
- The submission ID handling is already correct: the backend passes real database
  submission UUIDs as the id field and batch_analyze already returns them as
  submission_a and submission_b — do not change this

## Task 3 — Obfuscation-Aware Scoring (Design Before Code)

Files to read first: engine/api.py, engine/similarity/scores.py,
                     engine/obfuscation/detect.py

NOTE: Do not write any code for this task yet. Read and propose a design first.
No implementation until the design is reviewed and agreed on.

The current state:
The engine has two separate systems. The similarity scorer computes a score based on
how many k-gram fingerprints the two submissions share. It works on normalized tokens
where identifiers are replaced with ID and literals with NUM — this means basic
variable renaming is already invisible to the scorer. The obfuscation detector runs
separately after scoring and checks for structural tricks: loop type changes, dead
code insertion, reordering of code blocks, literal value substitution, wrapping in
try-catch blocks, and so on. It returns a list of flags but these flags currently
have no effect on the similarity score at all.

The problem:
A student who uses structural obfuscation — for example, swapping all for-loops to
while-loops and reordering a few methods — can meaningfully reduce the fingerprint
overlap even though the core logic is clearly copied. The similarity score goes down
because the fingerprint positions shift and fewer hashes match, even though the
actual plagiarism is still there. The professor sees a lower score and may not
investigate further. The obfuscation flags are shown but there is no connection
between them and the score.

What to think about before proposing a solution:
- The score must never be inflated blindly. If obfuscation flags are present but the
  raw fingerprint score is genuinely low, that may be a false positive from the
  obfuscation detector rather than confirmed plagiarism.
- The right signal might not be a boosted score but a separate adjusted score or a
  confidence indicator shown alongside the raw score.
- Consider: how many flags need to be present before the score is adjusted? A single
  flag (e.g. one renamed variable) is very different from five flags together.
- Consider: should the adjustment be proportional to the number of flags, or binary
  once a threshold of flags is crossed?
- Consider: the frontend currently shows similarityScore as a single number. Would it
  show two numbers? Or a label like "adjusted for obfuscation"?

After reading the code, propose a design. Explain what you would change, why, and
what risks exist. Do not write any code until the design is agreed on.
